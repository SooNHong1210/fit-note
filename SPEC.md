# 예약 앱 — 기획·워크플로우 문서 (v2, 결정 반영판)

> 개인 PT샵 / 회원제 예약 샵을 위한 가벼운 예약 서비스.
> 선생님이 싸게 쓰고, 회원은 가볍게 예약하며, 알림은 확실히 도달하는 것이 목표.

이 문서는 초기 기획서(P2P/E2E 안)를 5인 페르소나 검토를 거쳐 **서버 기반 단순 구조**로 재설계한 최종 결정판이다.

---

## 0. 핵심 방향 전환 요약 (왜 바뀌었나)

| 항목 | 초기 기획 | 최종 결정 | 이유 |
|---|---|---|---|
| 데이터 위치 | 선생님 폰에만 (E2E) | **서버 DB** | 데이터가 텍스트뿐이라 저장비 미미. E2E·P2P가 오히려 개발/운영비를 키움 |
| 통신 | P2P(WebRTC)+릴레이 | **서버 API** | TURN 등 변동비·복잡도 제거. 단순 CRUD가 더 쌈 |
| 인프라 | 자체 서버+릴레이 | **Supabase(서버리스, 유휴 0원)** | scale-to-zero, 표준 Postgres라 이전 가능(lock-in 회피) |
| 백업 | Stage3 유료 암호화 백업 | **서버 기본 제공** | 폰 분실=데이터 소멸 문제 자동 해소. 단 유료 백업 수익모델은 폐기 |
| 프라이버시 | 제품 정체성 | **나중에 옵션** | 민감필드 격리만 지금, 암호화 구현은 보류 |

## 1. 아키텍처

```
┌─────────────┐        ┌──────────────────────┐        ┌─────────────┐
│  회원 PWA    │  HTTPS │   Supabase            │  HTTPS │  선생님 PWA  │
│ (웹앱)       │◀──────▶│  - Postgres DB        │◀──────▶│ (웹앱)       │
│             │        │  - Auth               │        │             │
└─────────────┘        │  - Realtime(상태동기)  │        └─────────────┘
                       │  - Edge Function(알림) │
                       └──────────┬───────────┘
                                  │
                          알림 발송 (단계별)
                          - 초기: PWA 웹 푸시
                          - 이후: 카카오 알림톡(+SMS 폴백)
```

- **회원 / 선생님 둘 다 PWA(웹앱)**로 시작. 단일 코드베이스로 iOS·안드로이드·PC 커버.
- **데이터 접근은 레포지토리 계층 한 곳으로** 모은다 (DB 교체·암호화 도입 시 그 한 곳만 수정 → lock-in/부채 회피).
- 민감 가능성 있는 필드(수업 코멘트 등)는 **별도 컬럼/테이블로 격리**. 지금은 평문 저장, 나중에 그 필드만 암호화로 전환 가능하게.

## 2. 데이터 모델

회원 1명은 **하나의 기기 신원(Device)** 으로 **여러 샵(Membership)** 에 동시 소속될 수 있다. (멀티샵: 키 재생성 없이 샵별 가입 행만 추가)

```
Shop        { id, name, ownerDeviceId, businessHours, sessionMinutes, createdAt }

Device      { id, publicKey, createdAt }              # 회원 기기 = 영구 신원, 1개

Membership  { id, deviceId, shopId, memberName, birthDate,
              phone?,                                  # 알림톡용, 선택 입력
              status(active/pending/blocked), createdAt }
                                                       # 기기-샵 연결, 샵마다 1행

Lesson      { id, membershipId, startsAt, endsAt,
              status(scheduled/done/canceled) }

Comment     { id, membershipId, lessonId, text, createdAt }   # ← 격리 대상(민감 가능)

Booking     { id, membershipId, requestedSlot,
              status(requested/seen/approved/rejected),       # 상태 가시화용 4단계
              createdAt, respondedAt }

Availability{ id, shopId, weekday, startTime, endTime, slotMinutes }
```

> **격리 원칙:** `Comment.text`, `Membership.birthDate` 등 식별/민감 필드는 비즈니스 쿼리(WHERE/ORDER BY/JOIN)에 직접 엮지 않는다. 항상 "불투명해질 수 있는 형태"로만 다룬다.

## 3. 신원 · 로그인 워크플로우

### 결정 사항
- 본인확인은 **이름 + 생년월일** (가벼움 우선).
- 전화번호는 **알림톡 용도로만** 받음(선택). 인증 수단으로는 쓰지 않음.
- **선생님 승인 1탭**은 코드 자리만 잡아두는 **토글(기본 OFF)**. 사칭 문제(코멘트 열람)는 일단 열어둔 채 시작, 필요 시 ON.

### 3.1 선생님 온보딩
```
1. 샵 이름 입력 → Shop 생성
2. 기기 키쌍 생성 → ownerDevice 등록
3. 영업시간 + 세션 길이(예: 50분) 설정 → Availability 생성
4. 회원 추가: 이름 + 생년월일 입력 (전화번호 선택)
```

### 3.2 회원 최초 등록 (샵당 1회)
```
1. 샵 검색/선택
2. 이름 + 생년월일 입력
3. 기기에 키쌍 생성(최초 1회) → publicKey 등록
4. Membership 생성
   - 승인토글 OFF: status=active 즉시 사용
   - 승인토글 ON : status=pending → 선생님 폰 알림 → [승인] 탭 시 active
5. (선택) 알림톡 받을 전화번호 입력
```

### 3.3 이후 로그인
```
- 기기에 키 있음 → 자동 인증 (이름·생일 재입력 없음)
- 웹에서 캐시 삭제 등으로 키 없음 → 이름+생일 재확인 후 재등록(같은 흐름)
```

### 3.4 멀티샵
```
- 같은 기기로 B샵 가입 = 동일 publicKey로 Membership 행 추가
- 키 재생성 없음. 앱에 "내가 다니는 샵" 목록 표시
- 샵당 이름+생일 확인 1회는 정상 마찰 (각 선생님이 독립적으로 본인확인)
```

## 4. 예약 워크플로우

### 4.1 슬롯 계산 (서버)
```
세션길이 + 영업시간으로 슬롯 자동 생성
  └ 이미 잡힌 예약 제외
  └ (유료) 구글 캘린더 연동 시 외부 일정 제외
  → 회원에게 "예약 가능 슬롯"만 노출
```

### 4.2 예약 신청 → 승인 (상태 가시화 필수)
```
회원: 빈 슬롯 선택 → 신청
  Booking.status = requested
        │
        ▼  (선생님에게 알림 발송)
선생님 앱 열람 → status = seen   ← 회원 화면에 "선생님 확인 중" 표시
        │
        ▼
선생님 [승인]/[거절]
  approved → Lesson 생성, 회원에게 결과 알림
  rejected → 회원에게 결과 알림
```

- **회원 화면은 항상 현재 상태를 보여준다:** 신청됨 → 확인 중 → 승인/거절.
  알림을 못 받아도 앱을 열면 상태가 거기 있다(푸시 의존도↓).
- 선생님 오프라인이면 "아직 전달 전 — 보통 오전에 확인합니다" 식 **정직한 카피**. 빈 화면 금지.
- (미정) 24시간 무응답 처리: 조용한 자동취소 대신 "재알림 + 회원에게 계속 기다리기/취소 선택권".

### 4.3 수업 후
```
선생님: 수업 done 처리 → Comment 작성
회원: "내 기록"에서 예약 내역 + 코멘트 열람
```

## 5. 알림 전략 (단계별)

| 단계 | 선생님 알림 | 비고 |
|---|---|---|
| **초기(검증)** | PWA 웹 푸시 | 지인·안드로이드 위주 테스트엔 충분 |
| **유료 배포 전** | **카카오 알림톡 + SMS 폴백** | iOS PWA 푸시는 신뢰성 약함 → 실사용자 받기 직전 반드시 추가 |

- ⚠️ **마지노선:** 유료(돈 받는) 선생님에게 배포하기 전엔 알림톡/SMS를 붙인다. 알림 누락 = 신뢰 붕괴 = 이탈.
- 알림톡 준비물: 카카오 비즈니스 채널 + 사업자등록 + 템플릿 사전심사 + 대행사(솔라피/알리고 등). "알림톡 실패 → SMS 자동 대체" 설정.
- 회원 알림도 동일 채널 재사용.

## 6. 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 프론트 | **PWA** (회원+선생님) | 단일 코드베이스, 앱스토어/심사 회피, 개발비↓ |
| 백엔드/DB | **Supabase (Postgres)** | 무료 티어, 유휴 0원, 표준 SQL → 이전 가능 |
| 인증 | Supabase Auth + 기기 키 | 자동 로그인 |
| 실시간 | Supabase Realtime | 예약 상태 가시화 |
| 알림(초기) | Web Push (VAPID) | 무료 |
| 알림(이후) | 카카오 알림톡 + SMS 대행사 | 도달 보장 |
| 캘린더 연동(유료) | 구글 Calendar OAuth | 외부 일정 제외 |

> 프론트 프레임워크(React/Next/Svelte 등)는 미확정 — 착수 시 결정.

## 7. 수익 모델

**선생님 단위 프리미엄(Freemium) 구독.** 회원은 무료, 선생님(샵)이 결제.

| | 무료 | 유료 (₩9,900~19,900/월) |
|---|---|---|
| 회원 수 | ~15~20명 | 무제한 |
| 예약·달력·코멘트 | O | O |
| 알림 | PWA 푸시만 | **알림톡 자동 발송(+SMS 폴백)** |
| 구글 캘린더 연동 | X | O |
| 코멘트 히스토리 | 최근 일부 | 전체 |

- **유료 후크 = "확실한 알림 도달"**. 무료/유료 경계가 곧 변동비(알림톡)·고비용 기능과 일치 → 무료 사용자가 비용을 거의 안 일으킴.
- **가격 앵커링:** "카톡 무료 대비"가 아니라 **"노쇼 1건(≈5만원) 막으면 구독료 5배 회수"**.
- **단위경제:** 인프라 0 수렴 + 알림톡 변동비 월 ~1,700원/선생님 → 마진 80%+. ₩9,900에 팔아도 남음.
- **변동비 안전장치:** 유료 티어에 알림톡 월 할당량(예: 500건) 두고 초과 시 강등.
- **폐기:** 거래 수수료(결제 중개)·유료 암호화 백업(서버 기본 제공으로 명분 소멸)은 수익모델에서 제외.

## 8. 로드맵

- **Stage 0 — 선생님 도구 검증:** 선생님 PWA로 달력+회원관리+코멘트만. 통신·예약 없음. "쓸 만한가" 확인.
- **Stage 1 — 예약 루프(MVP):** 회원 PWA + Supabase + 예약 신청→승인 + 상태 가시화 + (초기)PWA 푸시. **여기까지가 MVP.**
- **Stage 2 — 알림 강화 + 유료화:** 카카오 알림톡, 구글 캘린더 연동, 구독 결제, 무료/유료 티어.
- **Stage 3 — 확장:** 공개 디렉터리/신규문의 전환, 게시판, 회수권/결제, (옵션) 민감필드 암호화를 세일즈 포인트화, 선생님 전용 얇은 네이티브(푸시 신뢰성 필요 시).

## 9. 착수 전 미해결 결정

1. **무료 티어 회원 상한** — 15명? 20명?
2. **유료 단가** — ₩9,900(박리다매) vs ₩19,900(기능 강조). → 사전 검증 후 결정.
3. **프론트 프레임워크** — React/Next/Svelte 등.
4. **예약 무응답(24h) 처리** — 자동취소 vs 재알림+선택권.
5. **사전 검증** — 선생님 5~10명에게 "이 기능에 월 1만원 낼 의사" 확인(빌드 전).

## 10. 설계 불변 원칙 (검토에서 못박은 것)

1. **알림은 반드시 선생님에게 도달** — 유료 배포 전 알림톡/SMS 이중화. 협상 불가.
2. **회원 예약 상태는 항상 가시화** — "선생님 미확인/오프라인"도 숨기지 말 것.
3. **유휴 0원** — 사용자 0명일 때 인프라비 0. 상시 VM 금지, 서버리스.
4. **민감필드 격리** — 첫날부터 추상화 경계 뒤로. 암호화 구현은 나중이어도 격리는 지금.
5. **이전 가능성** — 표준 Postgres + 레포지토리 계층으로 lock-in 회피.
